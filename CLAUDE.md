## Project Overview

Trae Account Manager is a Tauri 2 desktop app for managing multiple Trae IDE subscription accounts. It allows users to switch accounts, monitor token usage, manage machine IDs, and batch import/export accounts. The UI is in Chinese.

## Tech Stack

- **Frontend**: React 19 + TypeScript + Vite + Tailwind CSS v4
- **Backend**: Rust + Tauri 2
- **UI**: shadcn/ui (components), lucide-react (icons), recharts (charts)
- **Rust deps**: reqwest, tokio, warp (local HTTP server for login flow), serde, chrono
- **Platforms**: Windows, macOS

## Development Commands

```bash
# Install dependencies
npm install

# Run in dev mode (starts both Vite + Tauri)
npx tauri dev

# Build for production
npx tauri build

# Frontend only (Vite dev server, port 1420)
npm run dev

# Type check + build frontend
npm run build
```

## Architecture

### Two-process model (Tauri)

The app has a React frontend (`src/`) and a Rust backend (`src-tauri/`). They communicate via Tauri's command invocation system — frontend calls `invoke("command_name", args)` which maps to `#[tauri::command]` functions in Rust.

### Rust backend (`src-tauri/src/`)

- **`lib.rs`** — Main entry point. Defines `AppState` (wraps `AccountManager` in `Arc<Mutex<>>`), `ApiError` type, and all Tauri command handlers. Each command locks the mutex, delegates to `AccountManager`, and maps errors via `anyhow`.
- **`account/`** — Core business logic. `AccountManager` handles CRUD for accounts, token refresh, usage queries, import/export. `types.rs` defines `Account`, `AccountBrief` structs.
- **`api/`** — HTTP client for Trae's backend API. `trae_api.rs` makes reqwest calls; `types.rs` defines API response types.
- **`machine.rs`** — Reads/writes system machine IDs (Windows registry, macOS defaults). Also manages Trae IDE's machine ID and login state files.
- **`login.rs`** — Browser-based login flow. Spins up a local warp HTTP server on a random port, opens a Tauri webview to `trae.ai`, injects JS to hook fetch/XHR and intercept the `GetUserToken` response, then POSTs the captured token back to the local server.

### React frontend (`src/`)

- **`App.tsx`** — Root component. Manages account list state, view mode (grid/list), modals, and orchestrates all user interactions.
- **`api.ts`** — Thin wrapper around `@tauri-apps/api` `invoke()` calls. All frontend-to-backend communication goes through this file.
- **`types/index.ts`** — TypeScript interfaces mirroring Rust structs (`AccountBrief`, `UsageSummary`, etc.).
- **`components/ui/`** — shadcn/ui base components (Button, Dialog, Input, etc.). Generated by `npx shadcn add`.
- **`components/`** — App-specific components: `AccountCard`, `AccountListItem`, `Sidebar`, `AddAccountModal`, `DetailModal`, `UpdateTokenModal`, `ContextMenu`, `Toast`, `ConfirmModal`, `InfoModal`, `UsageEvents`.
- **`pages/`** — Top-level views: `Dashboard` (main account view), `Settings` (machine ID and Trae path config), `About`.
- **`hooks/useToast.ts`** — Toast notification state management.
- **`lib/utils.ts`** — shadcn `cn()` utility (clsx + tailwind-merge).
- **`index.css`** — Tailwind imports + shadcn CSS variables (oklch color scheme). No App.css — all styling is Tailwind utilities.

### Key patterns

- **Adding a new Tauri command**: Add handler in `lib.rs`, register in `generate_handler![]` macro, expose via `api.ts` on the frontend.
- **State access**: All commands receive `State<'_, AppState>`, lock the mutex, and call `AccountManager` methods.
- **Error handling**: Rust uses `anyhow::Result` internally, converts to `ApiError` (serializable) at the Tauri boundary. Frontend receives `{ message: string }` on errors.
- **Login flow**: The `login.rs` module is architecturally distinct — it creates a temporary warp server and a webview window, uses JS injection to intercept auth tokens, then closes both on success.
- **Adding shadcn components**: `npx shadcn@latest add <component-name>`. Components go into `src/components/ui/`.
- **Dialog pattern**: Uses shadcn v2 (`@base-ui/react`). Use `render` prop instead of `asChild` for composition (e.g., `DialogClose render={<Button />}`).
- **No test suite**: There are no frontend or backend tests. No ESLint/Prettier/Clippy config either.

### Account switching (most complex operation)

When `switch_account` is called in `machine.rs`: (1) kill Trae IDE process, (2) set machine ID from account's bound ID, (3) clear IDE state files (state.vscdb, Local State, IndexedDB, Local Storage, Cookies), (4) update storage.json with new telemetry/auth info, (5) reopen Trae IDE.

### Trae API endpoints

`TraeApiClient` in `api/trae_api.rs` calls multiple Trae backends:
- `https://api-sg-central.trae.ai` (Singapore, default)
- `https://api-us-east.trae.ai` (US East)
- `https://ug-normal.trae.ai` (user info)

Supports both cookie-based and JWT token auth. JWT payloads are base64-decoded client-side for user info extraction.

### Platform-specific code

`machine.rs` uses `#[cfg(target_os = "...")]` extensively. Windows uses registry (`winreg`) and `tasklist`/`taskkill`. macOS uses `ioreg`, `pgrep`/`pkill`, and `open -a`.

## Configuration

- **`vite.config.ts`** — Vite config with Tailwind CSS plugin and `@/*` path alias to `src/`.
- **`src-tauri/tauri.conf.json`** — Tauri app config (window size, bundle settings, dev server URL). Frontend dev server runs on port 1420.
- **`src-tauri/Cargo.toml`** — Rust dependencies. Note platform-specific deps: `winreg` (Windows), `core-foundation` (macOS).
- **`src-tauri/capabilities/`** — Tauri permission capabilities (default.json, login.json).
- **`components.json`** — shadcn/ui configuration (New York style, Zinc color, `@/*` import alias).
- **`tsconfig.json`** — TypeScript config with `@/*` path alias.
